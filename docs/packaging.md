# Packaging, reproducible builds & image transparency

SecGit is distributed as a single OSS build that runs **inside a confidential VM (CVM)**
on AMD SEV-SNP or Intel TDX. The public sandbox instance is that same build run in
sandbox mode — a configuration, not a fork. The trust claim ("the operator can't read
your code, and it's verifiable") rests on a clean chain:

```
source + Cargo.lock  ──reproducible build──▶  OCI image  ──launch──▶  SNP/TDX measurement
        │                                         │                          │
        └── SBOM (CycloneDX) ───────┐            └── image-manifest ─────────┤
                                     └────── transparency log (PQC-signed) ───┘
```

Anyone can correlate *what is running* (the SNP/TDX launch measurement obtained via
`/attestation`) with *what the source produces* (the published measurement in the
transparency log), and inspect the dependency graph (SBOM) and per-binary digests
(image manifest).

## Reproducible build

The image (`deploy/Dockerfile`) is built for determinism:

- **Pinned Rust toolchain** (`ARG RUST_VERSION`) and a pinned base image digest in CI.
- **`SOURCE_DATE_EPOCH`** is threaded into the build so timestamps are stable.
- **`cargo build --locked`** enforces `Cargo.lock`; no dependency may float.
- The runtime stage carries only the two static-ish binaries plus `git` and CA certs.

Build locally:

```bash
docker build -f deploy/Dockerfile \
  --build-arg RUST_VERSION=1.89.0 \
  --build-arg SOURCE_DATE_EPOCH=1700000000 \
  -t secgit/secgit-server:dev .
```

## Image-transparency artifacts

The build stage produces, deterministically:

- **`/usr/local/share/secgit/sbom.json`** — CycloneDX 1.5 SBOM generated by
  `cargo run -p xtask -- sbom Cargo.lock`. Every component carries its `pkg:cargo/...`
  purl and, for registry deps, the `Cargo.lock` SHA-256 checksum.
- **`/usr/local/share/secgit/image-manifest.json`** — SHA-384 digests of the
  `secgit-server` and `secgit-verify` binaries (`xtask measure`).

At runtime these are served publicly (they are not secrets):

| Route              | Content                                  |
| ------------------ | ---------------------------------------- |
| `GET /sbom`            | CycloneDX SBOM                        |
| `GET /image-manifest`  | per-binary SHA-384 digests           |
| `GET /attestation`     | fresh SNP report bound to the TLS key |

Generate the SBOM outside Docker too:

```bash
cargo run -p xtask -- sbom Cargo.lock sbom.json
```

## Publishing the launch measurement

The on-silicon launch measurement is computed with `xtask snp-measure` (wrapping
`sev-snp-measure` over OVMF + kernel + initrd + cmdline) and appended to the PQC-signed
transparency log via `xtask emit-transparency`. `Policy.allowed_measurements` is fed from
that log, so the server only accepts attestations matching a published OSS build. See
`docs/acceptance-snp.md` for the end-to-end on-hardware acceptance test.

## Runtime hardening (`deploy/docker-compose.yml`)

The container is locked down so that, even setting aside the CVM, the host-visible blast
radius is minimal:

- `read_only: true` root filesystem; writable state confined to the encrypted `/data`
  volume, with `tmpfs` for `/tmp` and the home dir.
- `cap_drop: [ALL]`, `no-new-privileges`, `pids_limit`.
- Runs as a non-root user (`uid 10001`).
- `configfs-tsm` (`/sys/kernel/config/tsm/report`) and `/dev/sev-guest` mounted for
  in-guest attestation; **no** cloud-attestation sidecar.
- `HEALTHCHECK` runs `secgit-verify selftest` (exercises the PQC primitives), not a
  plaintext probe.

The KEK is **never** baked into the image or compose file in production; it is released
by the attestation-gated key broker and lives only in CVM memory.
